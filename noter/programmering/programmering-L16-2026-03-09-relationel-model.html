Relationel Model og Datamanipulation

Programmering & Databaser · 9. marts 2026 · Ole Torp Lassen

Relationel model ER til RM Normalisering 1NF 2NF 3NF SQL CREATE TABLE INSERT UPDATE

Genereret 28. marts 2026

📋 Opsummering

Forelæsning 16 handler om at gå fra den konceptuelle ER-model til den logiske relationelle model og videre til en fysisk SQL-database. En relationel model er en samling af "flade" tabeller med navngivne kolonner og rækker, hvor alt data er atomisk og nøgler identificerer enhver række entydigt.

Konverteringen fra ER til RM følger faste regler: entiteter bliver tabeller, mange-til-mange relationer kræver en ny brygge-tabel, mens 1-til-mange løses med en fremmednøgle. Normalisering (1NF→2NF→3NF) sikrer at data er redundansfrit og fri for utilsigtede bi-effekter ved indsættelse, opdatering og sletning.

SQL-syntaksen til datamanipulation dækker CREATE TABLE med constraints, INSERT INTO, UPDATE og DELETE — de fire grundlæggende operationer der svarer til CRUD.

🔑 Vigtige begreber

Begreb Definition Eksempel / kontekst
Relationel model Logisk datamodel bestående af flade tabeller med navngivne kolonner og unikke rækker. dept(deptno, dname, loc)
Primærnøgle (PK) Kolonne(r) der entydigt identificerer hver række i en tabel. empno i emp-tabellen
Fremmednøgle (FK) Kolonne der peger på primærnøglen i en anden tabel — skaber relation mellem tabeller. deptno(FK) i emp peger på dept
Brygge-entitet En ny tabel der opløser en mange-til-mange relation. Indeholder FK fra begge sider + evt. egne attributter. project_members(empno, projectno, role)
1NF Første normalform: alle kolonner indeholder atomiske (udelelige) værdier — ingen lister i celler. En celle må ikke indeholde "Ole, Peter"
2NF Anden normalform: på 1NF + alle non-key attributter er fuldt afhængige af hele primærnøglen (ingen partielle afhængigheder). Hvis PK er (ono+partid), må pname ikke kun afhænge af partid
3NF Tredje normalform: på 2NF + ingen transitive afhængigheder — non-key attributter afhænger kun af PK. dname og loc afhænger af deptno, ikke empno → separat tabel
Transitivafhængighed A → B → C: C afhænger indirekte af PK via en anden non-key attribut. Brydes ved at splitte tabellen. empno → deptno → dname (bryd ud i dept-tabel)
Anomaly Utilsigtet bieffekt i ikke-normaliserede tabeller: update, insertion eller deletion anomaly. Sletning af Dr. Giddens fjerner kursusinformation
VARCHAR / INT / DATE SQL-datatyper: tekst med variabel længde, heltal og dato. ENAME VARCHAR(30), SAL INT

🎯 Til eksamen

  • ER til RM-reglerne: Entitet → tabel, mange-mange → ny tabel med sammensat PK, 1-mange → FK på mange-siden.
  • Normalformer: 1NF = atomiske værdier, 2NF = ingen partielle afhængigheder, 3NF = ingen transitive afhængigheder. Kend forskellen og giv eksempler.
  • Anomalies: Beskriv update, insertion og deletion anomaly — og hvorfor normalisering forhindrer dem.
  • CREATE TABLE med constraints: CONSTRAINT PK_emp PRIMARY KEY (empno) og CONSTRAINT FK_emp FOREIGN KEY (deptno) REFERENCES dept(deptno).
  • INSERT/UPDATE/DELETE-syntaks: Skriv alle tre uden at slå op.
  • NULL: Betyder "ikke defineret" — ikke 0 eller tom streng.
  • Datatyper: CHAR(n) fast længde, VARCHAR(n) variabel, INT, DATE, FLOAT.
📐 SQL-syntaks reference

CREATE TABLE med constraints

CREATE TABLE ILearn.EMP (
  EMPNO   INT         NOT NULL,
  ENAME   VARCHAR(30) NOT NULL,
  JOB     VARCHAR(10),
  MGR     INT,
  HIREDATE DATE,
  SAL     INT,
  DEPTNO  INT,
  CONSTRAINT PK_emp PRIMARY KEY (EMPNO),
  CONSTRAINT FK_emp FOREIGN KEY (DEPTNO)
    REFERENCES ILearn.DEPT(DEPTNO)
);

INSERT

INSERT INTO EMP (EMPNO, ENAME, JOB, SAL, DEPTNO)
VALUES (7369, 'SMITH', 'CLERK', 800, 20);

-- Kopi fra anden tabel
INSERT INTO OLDEMP (ENO, HDATE)
SELECT EMPNO, HIREDATE FROM EMP
WHERE HIREDATE < '1999-12-31';

UPDATE

UPDATE EMP
SET JOB = 'MANAGER', SAL = SAL + 1000
WHERE ENAME = 'JONES';

UPDATE EMP
SET SAL = SAL * 1.15
WHERE DEPTNO IN (10, 30);

DELETE

DELETE FROM PROJECT
WHERE BUDGET < 10000;

-- Slet hele tabellen
DROP TABLE EMP;

📖 Uddybning

Fra ER til relationel model — de fire trin

Trin 1 — Entiteter: Hver entitet bliver direkte til en tabel med samme attributter. Eks: emp(empno, ename, job, sal)

Trin 2 — Mange-mange: Opret en ny brygge-tabel med FK fra begge entiteter som sammensat PK, plus relationens egne attributter. Eks: project_members(empno, projectno, role)

Trin 3 — 1-mange: Placer en FK på mange-siden (eller lav en ny tabel hvis relationen har mange egne attributter). Eks: emp(empno, ..., deptno(FK))

Trin 4 — 1-1: Begge sider kan være mange-siden. Hvis én side er valgfri, placer FK hos den valgfri side. Eks: Car(licenseNo, ..., SSN)

Normalformer med eksempler

1NF — atomiske værdier: Hver celle må kun indeholde én værdi. Ingen lister, ingen gentagne grupper.

2NF — ingen partielle afhængigheder: Relevant ved sammensatte PK'er. Hvis en attribut kun afhænger af en del af PK'en, flyttes den til en separat tabel. Eks: order(ono, odate, partid) + part(partid, pname, price) — pname afhænger kun af partid, ikke af ono.

3NF — ingen transitive afhængigheder: Non-key attributter må kun afhænge af PK. Eks: emp-tabellen med dname og loc — de afhænger af deptno, ikke empno. Løsning: adskil i emp og dept-tabeller.

Anomalies — hvorfor normalisering er vigtigt

Update anomaly: Hvis samme data gemmes på flere rækker, kan en opdatering give inkonsistente data. Eks: medarbejder 519 har to forskellige adresser i samme tabel.

Insertion anomaly: Man kan ikke indsætte data uden at have alle påkrævede felter. Eks: Dr. Newsome kan ikke registreres som underviser, før han har fået tilknyttet et kursus.

Deletion anomaly: Sletning af en række fjerner utilsigtet anden information. Eks: Sletter man Dr. Giddens' kursusoplysning, mister man al info om ham.
