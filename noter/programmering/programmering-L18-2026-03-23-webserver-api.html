Webserver og API'er

Programmering & Databaser · 23. marts 2026 · Stefan

HTTP REST API Express.js Node.js CRUD JSON Client-Server Endpoints

Genereret 28. marts 2026

📋 Opsummering

Forelæsning 18 introducerer den tekniske rygrad i eksamensapplikationen: webserveren og API'et. En server stiller ressourcer til rådighed over et netværk, og HTTP er protokollen klient og server taler sammen med — én request giver ét svar.

Et Web API fungerer som en kontrolleret "dør" ind til systemet: klienten (browseren) kender kun API'ets endpoints, ikke den bagvedliggende logik eller database. Det sikrer adgangskontrol og separation of concerns.

I Node.js bygges serveren med Express, der lader én definere routes for hver HTTP-metode. CRUD-operationerne Create, Read, Update og Delete svarer direkte til POST, GET, PUT og DELETE — præcis det mønster eksamensopgaven kræver.

🔑 Vigtige begreber

Begreb Definition Eksempel / kontekst
HTTP HyperText Transfer Protocol — protokollen klient og server bruger til at kommunikere. 1 request = 1 svar. GET /pizza HTTP/1.1 → server svarer 200 OK
API Application Programming Interface — en mellemmand der lader programmer tilgå hinandens data kontrolleret, uden at kende den interne implementering. Flybetalingssiden kalder flyselskabernes API'er for at hente priser
Endpoint En specifik URL + HTTP-metode der udgør ét indgangspunkt i API'et. GET /todos, POST /todos, DELETE /todos/3
CRUD Create, Read, Update, Delete — de fire grundoperationer på data. Svarer til POST, GET, PUT, DELETE i HTTP
Express.js Node.js framework til at bygge webservere og API'er hurtigt og struktureret. npm install express, derefter app.get('/route', ...)
JSON JavaScript Object Notation — dataformat API'et bruger til at sende og modtage data. { "id": 1, "task": "Lær Express" }
Status code Talkode i HTTP-svaret der fortæller om requesten lykkedes. 200 OK, 201 Created, 404 Not Found, 500 Server Error
Separation of concerns Princippet om at holde database, API-logik og frontend adskilt — hver del har ét ansvar. Klienten kender ikke databasen, kun API-endpointene
Postman HTTP-testværktøj til at afprøve POST, PUT og DELETE endpoints. Send en POST-request med JSON-body til din lokale server
localhost:3000 Din lokale server kører på din egen maskine og lytter på port 3000. app.listen(3000) i Express

🎯 Til eksamen

  • HTTP-metoder og CRUD: GET=Read, POST=Create, PUT=Update, DELETE=Delete — lær det udenad.
  • Status codes: 200 OK, 201 Created, 404 Not Found, 400 Bad Request, 500 Internal Server Error.
  • Express-setup: npm init → npm install express → app.listen(3000).
  • Route-syntaks: app.get('/todos', (req, res) => res.json(todos)) — skriv det uden at slå op.
  • Middleware: express.json() er nødvendig for at læse JSON fra POST/PUT bodies.
  • Hvorfor API? Beskyttelse af databasen, adgangskontrol, separation of concerns — argumentér for dette i rapporten.
  • Eksamensarkitektur: Browser → Express API → SQL-database. Express er mellemleddet.
📐 Kode-snuttere

Minimal Express-server (index.js)

const express = require('express');
const app = express();
app.use(express.json()); // parse JSON body

app.get('/todos', (req, res) => res.json(todos));

app.get('/todos/:id', (req, res) => {
  const todo = todos.find(t => t.id == req.params.id);
  if (!todo) return res.status(404).json({ error: 'Ikke fundet' });
  res.json(todo);
});

app.post('/todos', (req, res) => {
  todos.push(req.body);
  res.status(201).json(req.body);
});

app.put('/todos/:id', (req, res) => {
  const idx = todos.findIndex(t => t.id == req.params.id);
  todos[idx] = { ...todos[idx], ...req.body };
  res.json(todos[idx]);
});

app.delete('/todos/:id', (req, res) => {
  todos = todos.filter(t => t.id != req.params.id);
  res.json({ message: 'Slettet' });
});

app.listen(3000, () => console.log('Kører på port 3000'));

HTTP Status Codes

// Succes
200 OK       → Request lykkedes
201 Created  → Ny ressource oprettet

// Fejl
400 Bad Request  → Forkert input fra klient
404 Not Found    → Ressourcen eksisterer ikke
500 Server Error → Noget gik galt på serveren

📖 Uddybning

Client-server arkitektur

Serveren er en enhed der deler ressourcer via et netværk. Klienten benytter disse ressourcer — det kan være en browser, en app eller et andet program.

Arkitekturen er typisk centraliseret: én server betjener mange klienter via HTTP — klienten sender en request, serveren svarer med et response. Én request giver præcis ét svar.

I eksamensopgaven: Browser → Express API → SQL-database. Browseren kender kun API-endpointene, ikke databasens indre.

Hvad er et API — og hvorfor bruger vi det?

API er en mellemmand der lader programmer tilgå hinandens data kontrolleret, uden at kende den interne implementering. Tænk på det som en restaurantmenu: du bestiller (request), køkkenet laver maden (logik du ikke ser), du får svaret (response).

  • Beskytter databasen — ingen direkte adgang udefra
  • Adgangskontrol — kontrol over hvem der kan se hvad
  • Separation of concerns — frontend, API og database er adskilte lag
  • Mindsker domæneviden — brugeren behøver ikke kende systemets indre
Express.js opsætning trin for trin
  • Lav en projektmappe og kør npm init -y — skaber package.json
  • Installer Express: npm install express
  • Lav index.js med din server-kode
  • Tilføj "start": "node index.js" under scripts i package.json
  • Start med npm start og gå til localhost:3000

Middleware express.json() er afgørende — uden det kan du ikke læse JSON-data fra POST/PUT request bodies.

Test af endpoints — browser vs Postman vs IDE

GET-endpoints kan testes direkte i browseren. Men POST, PUT og DELETE kan ikke testes i browseren — brug Postman eller REST Client i VS Code.

Postman: Vælg metode, skriv URL, tilføj JSON-body og send. Svaret vises med statuskode og body.

VS Code REST Client: Skriv .http-filer direkte i editoren og kør dem derfra. Praktisk til at gemme test-cases sammen med koden.

💻 Øvelse: Byg Todo-API'et

Trin 1 — Sæt projektet op

Lav en ny mappe, kør npm init -y og installer Express.

mkdir todo-api && cd todo-api
npm init -y
npm install express
Tilføj "start": "node index.js" i package.json under scripts.
Trin 2 — Sæt serveren op

Opret index.js. Serveren skal svare på GET / og lytte på port 3000.

const express = require('express');
const app = express();
app.use(express.json());

app.get('/', (req, res) => {
  res.send('Server kører!');
});

app.listen(3000, () => console.log('Kører på port 3000'));
Trin 3 — GET endpoints

Tilføj GET /todos og GET /todos/:id. Lav et lille array med 2-3 todos øverst. 

let todos = [
  { id: 1, task: 'Lær Express', done: false },
  { id: 2, task: 'Byg API', done: false }
];

app.get('/todos', (req, res) => res.json(todos));

app.get('/todos/:id', (req, res) => {
  const todo = todos.find(t => t.id == req.params.id);
  if (!todo) return res.status(404).json({ error: 'Ikke fundet' });
  res.json(todo);
});
Trin 4 — POST endpoint

Tilføj POST /todos der modtager en ny todo og svarer med 201.

app.post('/todos', (req, res) => {
  const nyTodo = { id: todos.length + 1, ...req.body };
  todos.push(nyTodo);
  res.status(201).json(nyTodo);
});
Test med Postman: POST til localhost:3000/todos, body: {"task":"Test","done":false}
Trin 5 — PUT og DELETE

Tilføj de to sidste endpoints.

app.put('/todos/:id', (req, res) => {
  const idx = todos.findIndex(t => t.id == req.params.id);
  if (idx === -1) return res.status(404).json({ error: 'Ikke fundet' });
  todos[idx] = { ...todos[idx], ...req.body };
  res.json(todos[idx]);
});

app.delete('/todos/:id', (req, res) => {
  todos = todos.filter(t => t.id != req.params.id);
  res.json({ message: 'Slettet' });
});
🏆 Bonus: Tilføj fejlhåndtering — hvad sker der hvis body er tomt ved POST? Returnér meningsfulde fejlbeskeder med korrekte statuskoder.
