OO+UML 2: Arkitektur og Design

Programmering & Databaser · 23. februar 2026 · Ole Torp Lassen

Arkitektur MVC 3-tier SSD DSD DCD OO Design Systemoperationer

Genereret 28. marts 2026

📋 Opsummering

Forelæsning 14 bygger videre på analysen fra L13 og bevæger sig over i design og arkitektur. Hvor analyse handler om at beskrive virkeligheden (domænet og problemet), handler design om at beskrive softwareløsningen — hvilke klasser, metoder og strukturer skal systemet have?

Systemsekvensdiagrammet (SSD) viser systemoperationerne fra brugerens perspektiv — hvad sker der når aktøren interagerer med systemet som en "black box". Arkitekturvalget (monolit, 3-tier eller MVC) afgør hvem der har ansvar for hvad, og danner grundlag for designsekvensdiagrammet (DSD), der folder systemet ud internt.

Designklassediagrammet (DCD) er det endelige resultat: klasser med attributter, metoder og relationer — klar til implementation. Processen er iterativ: én usecase ad gangen forfines fra SSD → arkitektur → DSD → DCD.

🔑 Vigtige begreber

Begreb Definition Eksempel / kontekst
SSD (Systemsekvensdiagram) Diagram der viser systemoperationer: aktørens inputs og systemets svar — systemet som black box. Aktør kalder submitTitleData() → system returnerer bekræftelse
DSD (Designsekvensdiagram) Folder systemet ud internt: viser hvilke klasser der kalder hinanden og med hvilke metoder. Controller → Model → Repository → Database
DCD (Designklassediagram) Det færdige klassediagram med attributter, metoder og relationer — grundlag for implementation. Klasse Kunde med attribut navn:String og metode getOrders()
Monolit Arkitektur hvor alt (UI, logik, data) ligger i ét lag. Simpelt men svært at skalere og vedligeholde. Én stor applikation uden lagdeling
3-tier Arkitektur med tre lag: præsentation (view), forretningslogik og datadatalag. Klienten kender ikke databasen. Browser → Server → Database
MVC Model-View-Controller: view viser data, controller håndterer input, model indeholder forretningslogik og data. Klient: View, Server: Controller + Model, DB: data
Systemoperation En operation som systemet tilbyder aktøren — identificeret via SSD. Bliver til metoder i controlleren. submitTitleData(title, date)
Repository Et lag der isolerer datadataadgang fra forretningslogikken. Controlleren kender ikke databasen direkte. Controller → Repositorium → Database
Creator/Knowledge Expert GRASP-principper: objektet der har data om noget, bør have ansvar for det. Den der skaber objekter bør kende dem. Ordre-klassen opretter Ordrelinjer (Creator)
Iterativ forfining Én usecase ad gangen: analyse → SSD → arkitektur → DSD → DCD → implementation → næste usecase. "One slice at a time"

🎯 Til eksamen

  • Analyse vs. design: Analyse beskriver virkeligheden (hvad systemet skal), design beskriver løsningen (hvordan systemet bygges).
  • SSD → DSD → DCD: Kend flowet og hvad hvert diagram viser. SSD = black box, DSD = intern interaktion, DCD = færdigt klassediagram.
  • Arkitekturerne: Monolit (alt ét sted), 3-tier (præsentation/logik/data), MVC (View/Controller/Model). Kend fordele og forskelle.
  • MVC i eksamensopgaven: Browser = View, Express = Controller, Node.js-klasser = Model, SQL = Database.
  • Repository-mønstret: Isolerer datadataadgang. Controller → Repository → Database — Controller kender ikke SQL direkte.
  • GRASP-principper: Creator (hvem opretter objekter?) og Knowledge Expert (hvem har ansvaret?) er de vigtigste.
  • Iterativ udvikling: Én usecase ad gangen forfines komplet — "one slice at a time".

📖 Uddybning

Fra SSD til DSD — fold systemet ud

SSD viser systemet som en sort boks: aktøren kalder en systemoperation (fx submitTitleData(title, date)), og systemet svarer. Det siger ikke noget om hvad der sker internt.

DSD folder den interne struktur ud: hvilke klasser involveres, i hvilken rækkefølge kaldes metoder, og hvad returneres? Arkitekturvalget styrer dette — i MVC vil Controller modtage kaldet, kalde Model, som igen kalder Repository, der tilgår databasen.

Huskeliste for DSD: livslinjer = klasser, beskeder = metoder på modtagerklassen, parametre/returværdier = metodesignaturer.

De tre arkitekturer sammenlignet

Monolit: Alt i ét. Simpelt at starte med, men svært at vedligeholde og skalere. Ingen klar adskillelse af ansvar.

3-tier: Tre fysisk adskilte lag: præsentation (browser/UI), forretningslogik (server) og data (database). Klienten kender ikke databasen. Giver bedre adskillelse og sikkerhed.

MVC: En konkretisering af 3-tier. View håndterer præsentation, Controller håndterer input og styrer flowet, Model indeholder forretningslogikken og datastrukturer. I en webapplikation: klienten er View, serveren deles i Controller og Model, databasen er datalagret.

I eksamensopgaven bruger I MVC: HTML/CSS = View, Express routes = Controller, Node.js-klasser = Model, SQL-database = persistence.

GRASP-principper — ansvarsfordeling

GRASP (General Responsibility Assignment Software Patterns) er principper for at afgøre hvilken klasse der skal have et givet ansvar.

Creator: Klasse A bør oprette objekter af klasse B, hvis A indeholder B, registrerer B, eller tæt bruger B. Fx: en Ordre bør oprette Ordrelinjer.

Knowledge Expert: Giv ansvaret til den klasse der har den nødvendige information. Fx: Ordrelinjen kender sin enhedspris, så den bør beregne sin delpris.

Man kan starte simpelt — "Controller skaber alt" — og gradvist forfine ved at flytte ansvar ud til de klasser der naturligt ejer det.

Repository-mønstret og persistenslaget

Et repositorium (repository) er et objekt der fungerer som en "samling" af domæneobjekter. Det isolerer al databaselogik fra resten af systemet.

Fordele: Controller og Model behøver ikke kende SQL. Hvis databasen skiftes ud, ændres kun repositoriet. Det gør koden nemmere at teste og vedligeholde.

I eksamensopgaven: Express-routes (Controller) kalder en Model-klasse, der kalder et Repository, der kører SQL mod databasen og returnerer objekter.
